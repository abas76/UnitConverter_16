## 2. 페르소나 [시뮬레이션]

**프로그래밍 수업에서 UnitConverter 부분 과제를 받은 2학년 전공 학생**

- 지난달 해외 직구(책상 받침대): 상품은 **인치**, 방 크기는 **센티미터** → 길이 변환 필요
- 과학 실험 보고서: **m / cm / mm** 혼용 금지 → 엑셀에서 단위 통일·재계산
- 변환 시 구글·네이버·유튜브·계산기·리뷰 사진 등 **여러 도구를 전환**하며 확인

## 3. Mom Test 증거

| # | 인용 (답변에서) |
|---|----------------|
| 1 | "두 번째는 **네이버 검색**도 열어 봤는데, 결과가 **소수점 자리가 조금 달라서 헷갈렸어요**." |

**행동 신호:** 2차 검색 → 수동 재계산 → 간접 검증(리뷰 사진) → 불확실성 속 구매 결정

## 4. 표면 문제 vs 진짜 문제

### 4.1 표면 문제 (잘못된 정의)

> "meter / feet / yard 길이 단위를 변환해 주는 Unit Converter 프로그램을 만든다."

- 겉으로는 "변환기가 없어서 불편하다"로 읽힘
- 인치↔센티 등 **실생활 단위**와 과제 단위(meter/feet/yard)가 **겹치지 않을 수 있음**
- "앱/프로그램을 만든다"에 집중하면 **신뢰·재확인 비용** 문제를 놓침

### 4.2 진짜 문제 (한 문장)

> 여러 곳에서 단위를 바꿀 때마다 **결과가 달라 신뢰할 수 없어**, 같은 값을 **다시 직접 계산하고 다른 자료로 교차 확인**해야 하며, 그 과정에서 **시간이 들고 구매·작업 판단이 흔들린다.**

| 차원 | 내용 |
|------|------|
| 불편 | 출처마다 결과가 달라 헷갈림 |
| 비용 | 여러 도구 전환, 수동 재계산에 시간 소모 |
| 판정 | 값을 믿지 못해 구매·제출 결정이 흔들림 |
| 재현 | 같은 입력인데 실행마다/출처마다 다른 값 |

### 4.3 과제 맥락과의 연결

| Mom Test 신호 | UnitConverter_16 대응 |
|--------------|----------------------|
| 결과 불일치 → 재계산 | 고정 변환 비율 + 테스트로 신뢰 확보 |
| 여러 도구 전환 | 한 입력 → 전 단위 일괄 출력 |
| 입력·형식 실수 | 음수·형식 오류·없는 단위 검증 |
| 단위가 상황마다 다름 | 동적 단위 등록 · 설정 외부화 (추가 요구사항) |

## 5. 주제 1문장

**한 번 넣은 길이 값이 meter / feet / yard로 항상 같은 기준으로 나오고, 잘못된 입력은 바로 걸러지게 한다.**

---

## 6. R-G-I-O

| 항목 | 내용 |
|------|------|
| **Role** | 길이 단위를 맞춰야 하는 사용자(학생·실습자)를 대신해 **일관된 변환 결과**를 보장하는 개발자 |
| **Goal** | 출처마다 다른 값·수동 재계산·교차 확인 없이, **한 입력으로 신뢰할 수 있는 전 단위 결과**를 얻고 판단 시간을 줄인다 |
| **Input** | `단위:값` (예: `meter:2.5`, `feet:8.2`, `yard:3`) — meter / feet / yard |
| **Output** | 동일 기준의 **전 단위 변환 결과** (예: `2.5 meter = 8.2 feet = 2.7 yard`) + 잘못된 입력 시 **명확한 오류** |

## 7. 성공 기준

| # | 기준 | Mom Test 증거 연결 |
|---|------|-------------------|
| 1 | **같은 입력 → 항상 같은 출력** (재검색·재계산 불필요) | "소수점 자리가 조금 달라서 헷갈렸어요" |
| 2 | **한 번 입력으로 meter / feet / yard 전부 확인** (여러 도구 전환 없음) | "2.54를 직접 곱해 봤습니다" |
| 3 | **형식 오류·음수·없는 단위는 즉시 차단** (우회 확인 전에 판정 가능) | "리뷰 사진 보면서 대략적인 크기 감 잡고" |

## 8. 이번 프로젝트에서 하지 않을 것

| 하지 않을 것 | 이유 |
|-------------|------|
| "변환 프로그램을 만든다"에만 집중 | 진짜 문제는 **도구 존재**가 아니라 **결과 신뢰·재확인 비용** |
| inch/cm, 해외 직구 등 페르소나 실생활 단위까지 기본 범위에 포함 | 기본 과제는 **meter / feet / yard** |
| GUI·웹·모바일 UI | CLI + 검증이 1차 목표 |
| Mom Test 없이 기능만 나열 | 요구사항 근거가 약해짐 |
