# PRD — UnitConverter_16

> Product Requirements Document  
> 근거: Mom Test 문제 정의 (`Report/01.UnitConverter_16_ProblemDefinition_Report.md`)

---

## 1. 제품 개요

사용자가 `단위:값` 형식으로 길이를 입력하면, **meter / feet / yard** 기준으로 **일관된 변환 결과**를 한 번에 출력하는 CLI 프로그램.

---

## 2. 문제 정의 (Mom Test 기반)

### 진짜 문제

여러 곳에서 단위를 바꿀 때마다 결과가 달라 신뢰할 수 없어, 같은 값을 다시 직접 계산하고 다른 자료로 교차 확인해야 하며, 그 과정에서 시간이 들고 구매·작업 판단이 흔들린다.

### 표면 문제 (지양)

"meter / feet / yard 변환 프로그램을 만든다" — 기능 존재만으로는 신뢰·재확인 비용을 해결하지 못함.

### 주제 1문장

**한 번 넣은 길이 값이 meter / feet / yard로 항상 같은 기준으로 나오고, 잘못된 입력은 바로 걸러지게 한다.**

---

## 3. R-G-I-O

| 항목 | 내용 |
|------|------|
| **Role** | 길이 단위를 맞춰야 하는 사용자를 대신해 일관된 변환 결과를 보장하는 개발자 |
| **Goal** | 출처마다 다른 값·수동 재계산·교차 확인 없이, 한 입력으로 신뢰할 수 있는 전 단위 결과 제공 |
| **Input** | `단위:값` 문자열 (예: `meter:2.5`) |
| **Output** | 전 단위 변환 결과 목록 + 입력 오류 시 명확한 메시지 |

---

## 4. 성공 기준

| # | 기준 | 검증 방법 |
|---|------|----------|
| SC-1 | 같은 입력 → 항상 같은 출력 | 변환 테스트 (재실행 일관성) |
| SC-2 | 한 입력으로 meter / feet / yard 전부 출력 | 통합 변환 테스트 |
| SC-3 | 형식 오류·음수·없는 단위 즉시 차단 | 입력 검증 테스트 |

**Mom Test 증거 연결**

- SC-1 ← "소수점 자리가 조금 달라서 헷갈렸어요"
- SC-2 ← "2.54를 직접 곱해 봤습니다"
- SC-3 ← "리뷰 사진 보면서 대략적인 크기 감 잡고"

## 5. 기능 요구사항

### 5.1 기본 (Must)

| ID | 요구사항 | 예시 |
|----|----------|------|
| FR-1 | `단위:값` 입력 파싱 | `meter:2.5` |
| FR-2 | meter / feet / yard 상호 변환 | `2.5 meter = 8.2 feet` |
| FR-3 | 한 입력 → 지원 단위 전체 출력 | meter, feet, yard 모두 |
| FR-4 | 입력 검증 | 음수, 잘못된 형식, unknown unit |
| FR-5 | 변환 정확도 테스트 | 고정 비율 기준 TC |

### 5.2 품질 (Must)

| ID | 요구사항 |
|----|----------|

| NFR-2 | SRP — 클래스별 단일 책임 |
| NFR-3 | 테스트 코드로 변환·검증 로직 검증 |

## 7. 입·출력 명세

### 입력

```
meter:2.5
feet:8.2
yard:3
```

### 정상 출력 (예)

```
2.5 meter = 2.5 meter
2.5 meter = 8.2 feet
2.5 meter = 2.7 yard
```

### 오류 출력 (예)

| 조건 | 메시지 |
|------|--------|
| `:` 없음 | `Invalid format. Use unit:value (ex: meter:2.5)` |
| 숫자 아님 | `Invalid number: {value}` |
| 없는 단위 | `Unknown unit: {unit}` |
| 음수 | (검증 후 오류 처리) |
